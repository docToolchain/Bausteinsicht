== Context

Risks captured during reviews need to be:

* expressed in a vocabulary the team already shares,
* tractable for AI-assisted code review (the project enables
  `risk-radar-skills@vibe-coding-risk-radar` in
  `.claude/settings.json`),
* aligned with the docToolchain ecosystem so risks can be aggregated
  across projects.

Candidates: *Vibe-Coding Risk Radar*, *OWASP Risk Rating*,
*CVSS-only*, *FAIR (Factor Analysis of Information Risk)*,
*ad-hoc free text*.

== Decision

*Vibe-Coding Risk Radar* — adopt the same taxonomy used by the
team's other tooling.

== Rationale

. *Already wired in.* `.claude/settings.json` enables the
  `risk-radar-skills@vibe-coding-risk-radar` plugin — risks identified
  during AI-assisted code review use the same vocabulary.
. *Aligned with docToolchain.* Bausteinsicht is part of the
  docToolchain ecosystem; reusing the taxonomy means risks can be
  aggregated across projects.
. *Lightweight.* No certification overhead, no quantitative model to
  parameterise — fits a small CLI's needs.

== Consequences

* Risks in `arc42/chapters/11_risks.adoc` use the radar's L/M/H
  severity scale and a short prefix (`Risk-E-*` for external,
  `Risk-S-*` for sync, `Risk-D-*` for documentation).
* New contributors should consult the radar plugin descriptions when
  adding risks.

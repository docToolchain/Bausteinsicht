= Acceptance Criteria
:toc: left
:toclevels: 2
:doctype: article

Gherkin-style scenarios derived from the test suite. Each scenario
references the underlying test name (in `*_test.go`) and the use case
ID from `01_use_cases.adoc`.

== UC-1 (`init`)

[source,gherkin]
----
Feature: Initialize a new project (UC-1)

  Scenario: Fresh directory
    Given an empty working directory
    When the user runs "bausteinsicht init"
    Then "architecture.jsonc", "template.drawio", "architecture.drawio"
         and ".bausteinsicht-sync" are created
    And the model file is valid
    And the draw.io file parses
    # Tests: TestInitCreatesFiles, TestInitGeneratesValidModel,
    # TestInitGeneratesValidDrawio, TestInitSyncStateValid

  Scenario: Refuse to overwrite
    Given "architecture.jsonc" already exists
    When the user runs "bausteinsicht init"
    Then the command exits with code 2
    # Test: TestInitFailsIfFilesExist

  Scenario: JSON output
    When the user runs "bausteinsicht init --format json"
    Then stdout contains a JSON object with the created file paths
    # Test: TestInitJSONOutput
----

== UC-2 (`sync`)

[source,gherkin]
----
Feature: Bidirectional synchronisation (UC-2)

  Scenario: After init, the very next sync is idempotent
    Given a project freshly created by "init"
    When the user runs "bausteinsicht sync"
    Then no changes are reported
    # Test: TestSyncAfterInit

  Scenario: Model changes propagate forward
    Given a synced project
    When the user edits a title in "architecture.jsonc"
    And runs "bausteinsicht sync"
    Then the corresponding draw.io shape's label is updated
    # Tests: TestSyncDetectsModelChanges, TestApplyForward_UpdatedTitle

  Scenario: draw.io edits propagate back to the model
    Given a synced project
    When the user changes a label in "architecture.drawio"
    And runs "bausteinsicht sync"
    Then the title in the JSONC model is updated
    # Tests: TestApplyReverse_TitleUpdate, TestApplyReverse_DescriptionUpdate,
    #        TestApplyReverse_TechnologyUpdate

  Scenario: User-added draw.io element is preserved
    Given a synced project
    When the user adds a shape in draw.io with a custom bausteinsicht_id
         that is not in the model
    And runs "bausteinsicht sync"
    Then the shape remains in the diagram
    # Test: TestSyncPreservesUserAddedDrawioElement

  Scenario: Conflicts resolve in favour of the model
    Given the same field has been changed in both the model and draw.io
    When the user runs "bausteinsicht sync"
    Then the model value wins
    And a conflict warning is reported
    And the exit code is 0
    # Tests: TestSyncConcurrentModelAndDrawioChanges,
    #        TestConflictResolverInterface

  Scenario: Validation errors abort the sync
    Given an invalid include pattern in a view
    When the user runs "bausteinsicht sync"
    Then the command exits with code 2 and no files are touched
    # Tests: TestSyncAbortsOnInvalidIncludePattern,
    #        TestSyncAbortsOnInvalidExcludePattern

  Scenario: A deleted draw.io file is recreated from the template
    Given the model exists but "architecture.drawio" was removed
    When the user runs "bausteinsicht sync"
    Then "architecture.drawio" is recreated
    # Tests: TestSyncRecreatesDeletedDrawioFile,
    #        TestSyncRecreatesDeletedDrawioFileWithCustomTemplate

  Scenario: An empty mxfile is recovered
    Given "architecture.drawio" is an empty mxfile
    When the user runs "bausteinsicht sync"
    Then a valid document is rebuilt
    # Test: TestSyncRecoversEmptyMxfile

  Scenario: Orphan view pages are removed
    Given a draw.io page "view-old" exists for a view that no longer
         appears in the model
    When the user runs "bausteinsicht sync"
    Then "view-old" is removed
    And pages whose IDs do not start with "view-" are kept
    # Test: TestSyncRemovesOrphanedViewPages

  Scenario: Comments in JSONC survive a roundtrip
    Given the model file contains line and block comments
    When the user runs "bausteinsicht sync" multiple times
    Then the comments are still present
    # Test: TestSyncPreservesJSONCComments

  Scenario: Verbose output goes to stderr
    Given "--verbose" is set
    When the user runs "bausteinsicht sync"
    Then progress lines appear on stderr (text format only)
    # Tests: TestSync_Verbose_PrintsSyncDetails,
    #        TestSync_Verbose_NotShownWithoutFlag,
    #        TestSync_Verbose_SuppressedWithJSONFormat
----

== UC-3 (`validate`)

----

== UC-4a (`add element`)

[source,gherkin]
----
Feature: Add a new element (UC-4a)

  Scenario: Top-level element
    Given a valid model
    When the user runs "bausteinsicht add element --id api --kind container --title 'API'"
    Then the JSONC file contains the new element
    # Test: TestAddElementTopLevel

  Scenario: Nested element under a container parent
    Given a system with container=true
    When "add element --parent shop --id api --kind container --title 'API'"
    Then "shop.children.api" appears in the JSONC
    # Test: TestAddElementNested

  Scenario: Reject duplicate ID
    # Test: TestAddElementDuplicateID, TestAddElementDuplicateNestedID

  Scenario: Reject parent that is not a container
    # Test: TestAddElementNonContainerParentRejected

  Scenario: Reject invalid IDs
    Given an ID containing dots, spaces, or starting with a digit
    Then the command exits with code 1
    # Tests: TestAddElementInvalidIDs, TestAddElementValidIDs

  Scenario: Comments are preserved
    Given a JSONC file with comments
    When an element is added
    Then no comment is lost
    # Test: TestAddElementPreservesComments
----

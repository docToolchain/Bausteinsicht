== 1. Problem statement

Architecture documentation tends to fall into one of two failure modes:

* *Visual-only* (draw.io, Visio, Lucidchart) — easy to read, easy to
  draft, but unstructured. There is no machine-readable model, so
  diagrams drift from reality and cannot be queried, validated, or
  generated by automation.
* *Code-only* (Structurizr DSL, LikeC4, custom DSLs) — structured,
  queryable, version-controllable, but architects must learn a new
  proprietary editor, and rendering is locked to the tool's renderer.

Bausteinsicht's thesis: *architects already use draw.io*, and giving
them a structured model alongside their existing visual editor — with
*bidirectional synchronisation* — removes the trade-off
_(Q-1.1.1, Q-1.3, OQ-6 team answer)_.

== 2. Vision

Architecture should be *as-code* without forcing teams to abandon
draw.io. A single JSONC text file is the source of truth; draw.io
provides the visual editor; both are reconciled on every `sync` cycle.
LLM agents become first-class clients _(Q-1.1.2, OQ-1 team answer)_.

== 3. Target users (personas)

_(OQ-1 team answer)_

[cols="1,3", options="header"]
|===
| Persona | Description

| Software Architect
| Designs and maintains the architecture model. Edits diagrams visually
in draw.io for layout decisions and `architecture.jsonc` for structure.
Primary author.

| Developer
| Consults diagrams and the model when implementing features. Adds
elements/relationships via `bausteinsicht add` from the terminal. Reads
the published HTML site (`gh-pages.yml`).

| LLM Agent (e.g., Claude, Kiro)
| First-class stakeholder. Uses `--format json` on every CLI command
to read and modify the model under the supervision of a developer or
architect. Skill files are shipped under `.claude/skills/bausteinsicht/`
and `.kiro/skills/bausteinsicht/`.
|===

Required context: the user is willing to use draw.io, and the team uses
Git for collaboration.

== 4. Jobs-to-be-Done

_(README, Q-1.1.2)_

. *Define architecture in text, get diagrams automatically.* User
writes JSONC and runs `bausteinsicht sync`; draw.io pages are created
and updated.
. *Edit diagrams visually, keep the model in sync.* User moves boxes
in draw.io; on next `sync` the JSONC reflects the changes.
. *Navigate multi-level views like a map.* Drill-down links connect
context → container → component pages; back-navigation buttons return
the user to the parent view _(Q-3.6.1; `internal/sync/forward.go:100-107,
168-170`)_.
. *Let AI agents modify the architecture.* Every command supports
`--format json` for deterministic, parseable output
_(`cmd/bausteinsicht/root.go:50`)_.

== 5. In-scope (v1) features

_(Q-2.1)_

* CLI commands: `init`, `sync`, `validate`, `add element`,
  `add relationship`, `watch`, `export`, `export-table`, `export-diagram`
  _(`cmd/bausteinsicht/root.go:55-62`)_.
* Bidirectional sync, model-wins conflict resolution
  _(`internal/sync/conflict.go:21-55`)_.
* Multi-view rendering, scope bounding boxes, relationship lifting,
  drill-down navigation _(`internal/sync/forward.go`)_.
* JSONC source format with comment preservation across edits
  _(`internal/model/loader.go:101-138`, `internal/model/patch.go`)_.
* JSON Schema-based IDE auto-completion
  _(`templates/sample-model.jsonc:2`)_.
* Cross-platform single binary (linux/darwin/windows × amd64/arm64)
  _(`Makefile:21-46`, `.goreleaser.yml`)_.

== 6. Out-of-scope (v1) — planned future considerations

_(OQ-2 team answer)_

* *As-Is / To-Be modelling* — render current and target architectures
  side by side.
* *Custom validation rules* — user-defined architectural constraints
  evaluated against the model.
* *PNG/SVG export pipelines* — the underlying `export` command exists
  _(`internal/export/export.go`)_, but a polished docs-pipeline workflow
  (e.g. Antora integration, asciidoctor includes) is not delivered as
  part of v1.
* *CI/CD architecture validation* — recipes/actions to gate PRs on
  validation results.
* *Structurizr / LikeC4 importers* — migration paths from existing
  text-based tools.

=== 6.1 Implemented but not yet exercised end-to-end

The `ConflictResolver` interface
_(`internal/sync/conflict.go:21-25`)_ is intentionally larger than the
single shipped implementation (`ModelWinsResolver`). This is *planned
extension*, not technical debt: the v1 strategy is documented and
sufficient _(OQ-3 team answer)_.

== 7. Non-goals

_(OQ-3 team answer)_

* *Multi-user real-time editing.* Bausteinsicht assumes one author at
  a time. Team collaboration is delegated to standard Git workflows
  (branch, edit, merge). The `.bausteinsicht-sync` file is a local
  cache, not a shared journal — it is single-machine
  _(`internal/sync/state.go:17-25`)_.
* *Network or server features.* No HTTP server, no outbound API
  calls, no TLS. Local CLI only _(OQ-7 team answer; `README.adoc:194`)_.
* *Lock-in to a custom editor.* The visual frontend is and remains
  draw.io _(OQ-6 team answer)_.

== 8. Success criteria

_(Q-4.2.2, OQ-8 team answer)_

[cols="1,3", options="header"]
|===
| Goal | Target

| Startup time
| < 10 ms (native Go binary).

| Sync time
| < 100 ms for models with up to 200 elements
(in-memory XML/JSON processing).

| Binary size
| 10–15 MB (single statically linked binary, CGO disabled).

| Dependencies
| Zero runtime dependencies.

| LLM ergonomics
| Every command supports `--format json` with stable schema
_(`cmd/bausteinsicht/root.go:18-26`)_.
|===

== 9. Constraints

_(Q-3.2)_

* Go 1.25+ required _(`go.mod:3`)_.
* AGPL-3 license _(`LICENSE`)_.
* All documentation in English, AsciiDoc format _(`CONTRIBUTING.md`)_.
* Direct dependencies limited to four:
  `beevik/etree`, `fsnotify/fsnotify`, `spf13/cobra`,
  `pgregory.net/rapid` _(`go.mod`)_.

== 10. Risks (summary)

_(Q-5; full list in `arc42/chapters/11_risks.adoc`)_

* draw.io schema drift is detected on load but not version-checked
  _(`internal/drawio/document.go:30-49`)_.
* `export --scale > 1` requires a hardware GPU and may fail silently in
  headless containers; mitigated with an output-file existence check
  _(`internal/export/export.go:56-66, 90-94`)_.
* The JSON Schema and Go `Element` struct are kept in sync manually;
  no CI assertion exists _(OQ-9 team answer)_.

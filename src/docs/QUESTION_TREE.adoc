== Q-1: What problem does this project solve and for whom?

=== Q-1.1: Who are the intended users?

==== Q-1.1.1: What user roles can be inferred from the code?

Status:: [ANSWERED]
Answer:: Three user roles are visible in the code surface:
* **Architect / engineer** — author of the JSONC architecture model and
  the draw.io diagram. Primary user of `init`, `sync`, `watch`, `add`,
  `validate`.
* **LLM / AI agent** — every CLI command supports `--format json` so a
  machine consumer can drive the tool programmatically.
* **Documentation reader** — recipient of `export`, `export-table`, and
  `export-diagram` outputs (PNG/SVG, AsciiDoc/Markdown, PlantUML/Mermaid).
Evidence::
* `cmd/bausteinsicht/root.go:50` — global `--format` flag with values
  `text` and `json`.
* `README.adoc:55-57` — "Let AI agents modify your architecture — every
  CLI command supports `--format json` for LLM-driven workflows."
* `cmd/bausteinsicht/export*.go` — `export`, `export-table`,
  `export-diagram` commands.

==== Q-1.1.2: Why was this project started? What alternatives existed?

Status:: [ANSWERED]
Answer:: _(team answer)_ Existing architecture-as-code tools
(Structurizr, LikeC4, Isoflow, c4model.com, C4InterFlow) do not
support draw.io — the most widely used free diagramming tool.
Bausteinsicht's core differentiator is bidirectional synchronization
with the tool architects already use, rather than forcing adoption of
a proprietary editor. ADR-001 evaluated 6 DSL options; JSONC scored
+20 (highest) because: no parser needed, bidirectional sync trivial,
universal IDE support via JSON Schema, LLM-native, comments via JSONC.
Evidence:: Team answer recorded 2026-05-02; reflected in
`src/docs/arc42/ADRs/ADR-001-DSL-Format.adoc` (Pugh Matrix) and
`src/docs/PRD/PRD-001.adoc` (Problem section).

==== Q-2.2.6: What include/exclude patterns are supported?

Status:: [ANSWERED]
Answer:: Five patterns:
* `id` — exact match.
* `prefix.*` — direct children of `prefix` only.
* `prefix.**` — all descendants (recursive).
* `*` — all top-level elements (no dot in ID).
* `**` — all elements.
Evidence::

are sanitised by lowercasing, replacing spaces with hyphens, and

Evidence::

=== Q-2.3: What are the use cases (Cockburn-style)?

==== Q-2.3.1: UC: Initialize an architecture project

Status:: [ANSWERED]
Primary actor:: Architect.
Trigger:: User runs `bausteinsicht init` in an empty directory.
Main success scenario:: System verifies no conflict files exist, writes
`architecture.jsonc`, `template.drawio`, `architecture.drawio`, and
`.bausteinsicht-sync`. Prints next-step guidance.
Failure modes:: At least one default file already exists (exit 2);
template parse error; sync state write error.
Evidence:: `cmd/bausteinsicht/init.go:33-128`.

==== Q-2.3.2: UC: Sync model and diagram (one shot)

Status:: [ANSWERED]
Primary actor:: Architect or LLM agent.
Trigger:: `bausteinsicht sync`.
Main success scenario::
. Auto-detect or use `--model`.
. Load model → validate → load draw.io → load state → load template.
. Compute three-way diff (model, draw.io, last-state).
. Resolve conflicts — model always wins (v1 strategy).
. Apply forward (model → drawio) and reverse (drawio → model) changes.
. Save model (comment-preserving patch where possible), drawio, and new state.
Failure modes:: Validation errors (exit 1); conflicts resolved but
exit 1; load/save errors (exit 2). If draw.io file is missing or has
no `<diagram>` elements, it is recreated and state is reset.
Evidence::
* `cmd/bausteinsicht/sync.go:31-203`.
* `internal/sync/engine.go:31-93` — pure pipeline.

==== Q-2.3.3: UC: Continuously sync on save (watch mode)

Status:: [ANSWERED]
Primary actor:: Architect.
Trigger:: `bausteinsicht watch`.
Main success scenario:: Both files exist → fsnotify watcher started →
on each write/create event, debounce `300ms`, run a sync, print summary
to stdout.
Failure modes:: Either input file missing at start (exit 2); transient
errors during a sync are logged but do not stop the watcher.
Evidence:: `cmd/bausteinsicht/watch.go:28-184`,
`internal/watcher/watcher.go:78-173` (re-watch on rename).

=== Q-3.4: arc42 §4 — Solution strategy

==== Q-3.4.1: What is the high-level solution approach?

Status:: [ANSWERED]
Answer::
* Source of truth is a JSONC model file. JSONC enables author comments
  in addition to data.
* The diagram (`architecture.drawio`, mxfile XML) is a *projection*,
  not a separate source of truth — but it is also editable.
* A persisted `.bausteinsicht-sync` state file provides the third leg
  for three-way diff (model, drawio, last-known).
* Sync is a *pure function* (`internal/sync/engine.go:Run`) on
  `(model, doc, lastState, templates)` returning a `SyncResult`. All
  I/O is at the edges (CLI command).
* On conflict, the model always wins (v1 strategy, pluggable interface).
Evidence::
* `internal/sync/engine.go:21-93` — pure `Run`.
* `internal/sync/conflict.go:23-55` — `ConflictResolver` interface +
  `ModelWinsResolver`.

=== Q-3.6: arc42 §6 — Runtime view

==== Q-3.6.1: What is the runtime sequence of `sync`?

Status:: [ANSWERED]
Answer::
. CLI parses flags, validates `--format` and path containment.
. `model.Load` → strip JSONC comments → unmarshal → extract element
  ordering for layout layers.
. `model.Validate` → list of errors; non-empty fails fast.
. `drawio.LoadDocument` (or recreate from template if missing/empty).
. `sync.LoadState` (verifies SHA-256 checksum; missing/zero-length →
  empty).
. `drawio.LoadTemplate(FromBytes)` for shape styles.
. Ensure a draw.io page exists for every view; track newly added pages.
. `sync.RemoveOrphanedViewPages` deletes pages for deleted views.
. `sync.Run`:
  - `DetectChanges` — three-way diff into a `ChangeSet`.
  - Conflict resolution (model wins) → drop conflicting drawio changes.
  - `ApplyForward` (model → drawio).
  - `ApplyReverse` (drawio → model).
  - Collect warnings (invisible elements, conflict warnings).
. Save model (`PatchSave` if simple field change; full `Save` otherwise).
. `drawio.SaveDocument` (atomic temp + rename).
. `sync.BuildState` + `sync.SaveState` (atomic, checksummed).
. Print summary; non-zero exit if conflicts.
Evidence::
* `cmd/bausteinsicht/sync.go:31-203`.
* `internal/sync/engine.go:21-93` — phases enumerated in the doc comment.
* `internal/model/loader.go:17-46`,`156-230` — JSONC stripping.
* `internal/sync/state.go:46-93` — checksum verification.

==== Q-3.6.2: What is the runtime sequence of `watch`?

Status:: [ANSWERED]
Answer:: Watch starts an fsnotify watcher on `architecture.jsonc` and
`architecture.drawio`; on each write/create the callback debounces
`300ms`, then runs the same sync pipeline as `sync` (without
`--relayout`). The `syncing` flag suppresses re-entrant events caused
by sync's own writes. Atomic-rename writes (delete-then-create from
editors) trigger a `rewatch` goroutine that polls for the file and
re-adds it.
Evidence::
* `cmd/bausteinsicht/watch.go:28-87,89-184`.
* `internal/watcher/watcher.go:64-76,90-131,133-173`.

==== Q-3.6.3: How is conflict resolution implemented at runtime?

Status:: [ANSWERED]
Answer:: A `Conflict` is recorded only when *both* sides changed the
same `(elementID, field)` triple since the last sync. The
`ModelWinsResolver` produces a `ResolvedConflict{Winner: "model",
Warning: …}` for each. Conflicting draw.io-side changes are then
filtered out of `DrawioElementChanges` before reverse sync is applied.
Evidence::
* `internal/sync/diff.go:518-526,652-663` — `checkElemConflict`.
* `internal/sync/conflict.go:23-55` — `ModelWinsResolver`.
* `internal/sync/engine.go:46-56,124-153` — filter step.

=== Q-3.8: arc42 §8 — Cross-cutting concepts

==== Q-3.8.1: How are JSONC comments preserved across saves?

Status:: [ANSWERED]
Answer:: Two paths:
* `PatchSave` (`internal/model/patch.go`) replaces only the JSON values
  for a list of `(path, value)` operations, leaving comments and key
  ordering intact. Used when the reverse sync produces simple
  field-modifications (`bsync.ReversePatchOps` returns `patchable=true`).
* If patching fails or the change is structural, `model.Save` falls
  back to a full re-marshal that preserves the file's *preamble*
  (everything before the root `{`) — capturing top-of-file comments.
Evidence::
* `internal/model/patch.go:9-50`.
* `internal/model/loader.go:101-138,232-245` — preamble preservation.
* `cmd/bausteinsicht/sync.go:243-265` — Patch-then-fallback policy.

==== Q-3.8.2: How is the persisted sync state protected?

Status:: [ANSWERED]
Answer:: `SyncState` is JSON with a SHA-256 checksum. On save, the
state is marshalled twice — once without checksum to compute the hash,
once with it included. On load, the checksum is recomputed and a
mismatch returns a corruption error directing the user to delete
`.bausteinsicht-sync` and re-sync. A zero-byte file is treated as
"no state" (e.g. interrupted write).
Evidence:: `internal/sync/state.go:46-132`.

==== Q-3.8.3: How are file writes made durable?

Status:: [ANSWERED]
Answer:: All persistent files (`*.jsonc`, `*.drawio`, sync state) use
`os.CreateTemp` in the destination directory + `os.Rename` after a
successful write. This avoids TOCTOU races and partial writes.
Evidence::
* `internal/model/loader.go:118-137`.
* `internal/drawio/document.go:55-79`.
* `internal/sync/state.go:96-132`.

==== Q-3.8.4: How are draw.io elements correlated with model elements?

Status:: [ANSWERED]
Answer:: Each draw.io `<object>` carries `bausteinsicht_id`
(canonical model dot-path) and `bausteinsicht_kind` attributes.
When views are used, the *cell* `id` is scoped (`<viewID>--<elemID>`)
to allow the same element to appear on multiple pages, while the
`bausteinsicht_id` remains canonical.
Evidence::
* `internal/sync/diff.go:203-236` — extraction.
* `internal/sync/diff.go:296-314` — `stripScopedPrefix` and
  `resolveCellID`.

==== Q-3.8.5: How are unmanaged user-drawn shapes handled?

Status:: [ANSWERED]
Answer:: A `<object>` without `bausteinsicht_id` that wraps a vertex
mxCell is treated as an "added element" by reverse sync. The element
ID is derived from the label text via `sanitizeID` (lowercase, dashes,
strip `. / \`). Navigation buttons (`nav-back-…`), metadata, and legend
boxes are excluded.
Evidence:: `internal/sync/diff.go:241-294`.

==== Q-3.8.6: How is auto-layout produced?

Status:: [ANSWERED]
Answer:: `internal/sync/layout.go` computes element positions on a
fresh page using A4-landscape dimensions (`pageWidth=1169`,
`elementGap=60`, `padding=60`). The layer order for "layered" layout
is taken from the user's order of element kinds in
`specification.elements` — captured by streaming JSON tokens at load
time (`extractElementOrder` in `loader.go`). Three explicit layouts:
`layered`, `grid`, `none` (and empty default).
Evidence::
* `internal/sync/layout.go:12-50`.
* `internal/model/loader.go:43,51-99` — element-order extraction.
* `internal/model/validate.go:167-172` — `validLayouts`.

==== Q-3.8.7: How does multi-view drill-down navigation work?

Status:: [ANSWERED]
Answer:: For each view that has a `scope`, the scoped element is
endowed with a draw.io `link` attribute of the form
`data:page/id,view-<viewID>`. Detail-view pages get a back-navigation
button (`nav-back-…`) added by `createBackNavButton`. These are
ignored by reverse sync.
Evidence::
* `internal/sync/forward.go:101-107,160-170`.

=== Q-3.12: arc42 §12 — Glossary

==== Q-3.12.1: Which domain terms appear in the code with definitions?

Status:: [ANSWERED]

=== Q-5.5: Operational risks

==== Q-5.5.1: What can silently break a project's history?

Status:: [ANSWERED]

draw.io exits 0 with no output. Mitigation: `BuildExportArgs` only
passes `--scale` when `>1` (a partial mitigation; users who set
`--scale 2` still hit it). Additionally, post-export the file
existence is verified.
Evidence:: `internal/export/export.go:58-69,84-96`.

Status:: [ANSWERED]
